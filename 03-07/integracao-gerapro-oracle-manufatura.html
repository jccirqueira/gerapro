Integração GeraPro → Oracle Manufacturing

Análise técnica para alimentar o ERP Oracle com dados do GeraPro a partir do fechamento de uma proposta técnica

03 de Julho de 2026 • Cliente: Drivetech

◆ Contexto

A Drivetech está implantando o ERP Oracle com o módulo de Manufacturing (Oracle Cloud SCM / E-Business Suite) e continuará usando o GeraPro para a fase de engenharia de propostas técnicas.

O ponto de integração natural é o fechamento (won) de uma proposta — todo o dado de engenharia gerado no GeraPro precisa ser consumido pelo Oracle para iniciar a produção dos painéis elétricos.

Objetivo Eliminar a redigitação manual entre sistemas, garantindo que itens, BOMs e ordens de produção criados na engenharia fluam automaticamente para o Oracle Manufacturing.

◆ Data Mapping: GeraPro → Oracle

Item Master (Materiais)

Cada material único do catálogo do GeraPro utilizado na proposta vira um item no Oracle.

GeraPro Oracle Tabela Coluna Oracle Exemplo
materiais[].codigoFabricante MTL_SYSTEM_ITEMS segment1 (Item Number) 3RT2017-1AP00
materiais[].descricao MTL_SYSTEM_ITEMS description Contator Siemens 22A 220V
materiais[].fabricante MTL_SYSTEM_ITEMS manufacturer_part_number Siemens
materiais[].unidade MTL_SYSTEM_ITEMS primary_uom_code EA
materiais[].categoria MTL_SYSTEM_ITEMS Item Category (flexfield) Contator
materiais[].custo CST_ITEM_COSTS item_cost 150.00
materiais[].ncm MTL_SYSTEM_ITEMS hs_code 8536.50.00

Bill of Materials (Estrutura dos Painéis)

Cada equipamento/painel vira um Assembly no Oracle, e seus materiais associados viram Componentes do BOM.

GeraPro Oracle Tabela Mapeamento
equipments[].tag BOM_BILL_OF_MATERIALS Assembly Item (ex: CCM-01)
equipments[].materials[] BOM_COMPONENTS Componente + Quantidade
equipments[].automationMaterials[] BOM_COMPONENTS Componentes derivados da I/O List
loads[] + typical.items[] BOM_COMPONENTS Componentes dos típicos (resolvidos via BOM do típico)
infraestrutura.disciplinas[].itens[] BOM_COMPONENTS ou WIP Materiais de infraestrutura (obra civil, cabos, etc.)

Work Orders (Ordens de Produção)

Cada equipamento gera uma WO no Oracle para fabricação do painel.

GeraPro Oracle Tabela Mapeamento
equipments[].tag WIP_DISCRETE_JOBS.wip_entity_name Nome da WO = Tag do equipamento
equipments[].type WIP_DISCRETE_JOBS Classificação (CCM-BT, QGBT, Cubículo MT, etc.)
quantidade (sempre 1) WIP_DISCRETE_JOBS.quantity 1 painel
revisions[].data WIP_DISCRETE_JOBS.date_released Data de liberação
projeto + codigo WIP_DISCRETE_JOBS Referência cruzada (proposal number)

◆ Abordagem A — FBDI (File-Based Data Import) Recomendado para MVP

Oracle fornece planilhas Excel padronizadas (FBDI) para carga de dados em lote. É o método mais usado, auditável e compatível com qualquer versão do Oracle (Cloud ou EBS).

Fluxo GeraPro (proposta ganha) → Botão "Exportar para Oracle" → Gera 3 arquivos .xlsx no formato FBDI → Usuário baixa e importa no Oracle via File Import → Oracle valida e processa

Materiais únicos extraídos do catálogo da proposta (combinando equipments[].materials[], automationMaterials[], infraestrutura[].itens[] e loads → typicals).

Prós & Contras

Prós Contras
Simples de implementar (Excel + template) Processo manual (upload no Oracle)
Compatível com Oracle Cloud e EBS Batch — não é tempo real
Auditável — arquivo pode ser versionado Download + upload externo ao sistema
Não requer credenciais Oracle no GeraPro Erro de validação só descoberto no upload
Qualquer usuário pode executar Depende do usuário lembrar de exportar

◆ Abordagem B — REST API (Oracle Cloud) Avançado

Oracle Cloud SCM oferece REST APIs públicas. Um serviço de integração (Node.js separado ou módulo no próprio GeraPro) chama as APIs diretamente quando a proposta é ganha.

APIs Oracle Necessárias

Oracle REST Resource Método Finalidade
/fscmRestApi/resources/11.13.18.05/items POST Criar Item Master
/fscmRestApi/resources/11.13.18.05/boms POST Criar BOM com componentes
/fscmRestApi/resources/11.13.18.05/wipJobs POST Criar Ordem de Produção
/fscmRestApi/resources/11.13.18.05/salesOrders POST Criar Pedido (opcional)

Arquitetura do Integration Service

O fluxo completo da integração via REST API, desde o fechamento da proposta até o Oracle:

1
✓
Proposta é fechada (won)
Usuário move o pipeline para "Fechado" ou clica em "Exportar para Oracle"
Aciona: webhook /api/integration/oracle/export
2
🗃
Coleta dados da proposta
Integration Service carrega o JSON completo da proposta e o catálogo de materiais
Fontes: activeTechnicalProposal + store.state.materiais
3
⚙
Transforma para payload Oracle REST
Converte a estrutura do GeraPro no formato esperado pelas APIs Oracle
Mapping: equipments[].tag → assembly_item, materials[] → components[]
4
☁
Envia para Oracle via REST
Chama sequencialmente as APIs: Itens → BOMs → Work Orders
Autenticação: OAuth 2.0 • Retry: 3 tentativas com backoff
5
📊
Log + Dashboard de Integração
Registra status de cada etapa e disponibiliza dashboard no GeraPro
Campos: proposta_id, data, usuario, status, payload, erro

Prós & Contras

Prós Contras
Tempo real — proposta ganha = itens criados Depende de Oracle Cloud (não funciona com EBS on-premise clássico)
Zero intervenção manual Requer setup de OAuth / credenciais Oracle
Log e rastreabilidade automáticos Mais complexo de implementar e testar
Pode ser expandido para outros módulos (Financeiro, Supply Chain) Tratamento de erros e retry precisa ser robusto
Dashboard de status da integração Manutenção contínua (versões da API Oracle)

◆ Abordagem C — Híbrida Mais indicada para o cenário

Dado que a Drivetech está implantando o Oracle (ou seja, ainda definindo processos e sem APIs em produção), a abordagem híbrida é a mais profissional e realista.

⏱ Fase 1 — Agora (MVP) Já pode começar
Criar o módulo de exportação FBDI no GeraPro
Entrega valor em dias, não em meses. A equipe Oracle da Drivetech pode começar a testar a importação imediatamente.
  • Botão "Exportar para Oracle" na proposta ganha
  • Coletor inteligente de materiais (unifica todas as fontes)
  • Gerador dos 3 arquivos FBDI (.xlsx)
  • Log de exportação (quem, quando, qual proposta)
  • Configuração de mapeamento (categoria → item type)
GeraPro → .xlsx → Oracle
⏲ Fase 2 — 3 a 6 meses Quando Oracle estável
Criar o Integration Service REST
Quando o ambiente Oracle estiver estável e as APIs definidas, automatizar completamente.
  • Microserviço Node.js separado (ou módulo no GeraPro)
  • Autenticação OAuth 2.0 com Oracle Cloud
  • Chamadas REST para Itens, BOMs e WOs
  • Webhook no GeraPro para disparo automático
  • Dashboard de status com histórico e retry
GeraPro → REST → Oracle
Roadmap Visual As duas fases podem coexistir. O fluxo FBDI continua funcionando mesmo depois do REST implementado — servindo como fallback manual.

◆ Considerações Finais

  • O GeraPro não precisa se conectar ao Oracle — ele apenas gera dados estruturados no formato que o Oracle espera consumir.
  • O formato FBDI é padrão Oracle — qualquer implementação Oracle (Cloud ou EBS) suporta. Não há risco de incompatibilidade.
  • A Drivetech está em fase de implantação — começar com FBDI permite que a equipe de Oracle da Drivetech valide o mapeamento antes de automatizar via API.
  • O Integration Service REST pode ser desenvolvido posteriormente sem quebrar o fluxo FBDI — ambos podem coexistir.
  • Manter o catálogo de materiais sincronizado é o ponto mais crítico — o codigoFabricante no GeraPro deve ser o mesmo item number no Oracle.
Próximo passo recomendado Validar com a equipe Oracle da Drivetech: (1) qual versão do Oracle (Cloud ou EBS)? (2) quais os templates FBDI oficiais que eles usam? (3) há algum campo customizado no Item Master que precisamos mapear?
